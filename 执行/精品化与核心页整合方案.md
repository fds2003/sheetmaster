## 一、统一策略与原则

| 原则 | 说明 |
|------|------|
| **质量大于数量** | 约 50 个核心页，每页「精品」；不自动批量生成列表外页面。 |
| **一个关键词组 = 一个页面** | 同义词/近义词合并到同一 URL（如 vlookup generator/syntax/tool → /formulas/vlookup），避免内竞。 |
| **每页须有独有元素** | 独有 Meta、独有输入参数、独有 FAQ Schema、独有错误排查（Common Errors）；禁止「只换公式名」的模板。 |
| **新增页面流程** | 新页须走 Boutique 流程：定义独有参数 → 撰写独有 FAQ → 撰写独有场景案例。 |

## 三、每页质量标准（精品化要求）

### 3.1 所有公式页（含 Hero + Essential）

| 项目 | 要求 |
|------|------|
| **Title / Meta Description** | 独有表述，体现该函数用途或痛点；禁止仅替换函数名的套话。 |
| **OG / Twitter description** | 与 `metaDescription` 一致（或等价短句），避免全站分享文案雷同。 |
| **输入参数** | 每公式在 `lib/formulas.ts` 有独立 `inputs[]`，交互按函数定制。 |
| **FAQ Schema** | Hero 10 必填；Essential 按需。问题须为**该公式特有**（如 VLOOKUP: Why #N/A?）；禁止「什么是 X」类通用问。 |
| **Common Errors & Fixes** | 每公式页有独立区块，内容按公式写（VLOOKUP：格式不匹配、列号错误；INDEX MATCH：范围高度不一致等）。 |
| **场景化教程** | Hero 10 须有约 800–1000 字 richContent（场景+案例+可选截图）；Essential 可选。 |

### 3.2 首页、Solutions、Use-cases、Compare

- **首页**：关键词与 USP 见 2.1；Meta/OG 与定位一致。  
- **Solutions**：每页独有 title/description、核心关键词、工具特性见 2.3。  
- **Use-cases**：痛点 + 推荐公式/工具 + CTA；独有 meta。  
- **Compare**：独有 title/description，正文针对对比关键词。
