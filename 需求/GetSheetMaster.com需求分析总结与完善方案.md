## 二、主要问题诊断

| 问题 | 表现 | 影响 |
|------|------|------|
| **无单一入口** | 没有「分析/策略总索引」或 README，新人不知道先看哪份 | 上手成本高，重复阅读 |
| **总览文档多头** | 网站分析概述、网站分析与优先行动建议、流量提升策略总结、GROWTH_STRATEGY_REVIEW 等 4–5 份都做“总结+建议” | 结论相似，维护分散，易产生版本不一致 |

## 三、完善方案总则

### 3.1 目标

1. **一入口**: 通过一份「分析/ 使用说明 + 总索引」快速定位策略与执行文档。
2. **单一事实来源**: 现状诊断、核心结论、优先级只维护在少数指定文档中，其余文档引用之。
3. **分层清晰**: 策略层（为什么/做什么）与执行层（怎么做/Prompt/代码）分目录或分文件，便于按角色使用。
4. **可追溯**: 关键词清单、执行方案保留历史版本时注明日期与版本说明，并明确“当前主版”。

### 3.2 原则

- **合并优于新增**: 优先合并同质文档，减少“又一版总结”。
- **引用优于复制**: 能引用已有文档的结论或清单时，不重复粘贴大段内容。
- **命名统一**: 项目名统一为 SheetMaster，产品/域名用 getsheetmaster.com；文档名以中文为主，必要时括号英文。

## 七、总结

| 维度 | 现状 | 目标 |
|------|------|------|
| **入口** | 无统一入口，文档多而散 | 一份 README + 明确主策略/主清单 |
| **现状与策略** | 4–5 份总览结论重复 | 1 份现状 + 1 份阶段路线图，其余引用或归档 |
| **关键词** | 10+ 份清单，版本不清 | 1 份主清单 + 明确主版/历史/补充 |
| **执行与 Prompt** | 多份“可直接执行”交叉重复 | 按类型合并 + 统一 Prompt 索引 |
| **命名与规范** | 中英混用、项目名不统一 | 统一 SheetMaster / getsheetmaster.com，新文档规范命名 |
| **与项目其他文档** | 未显式对齐 | 与 openspec、PRD、SEO_OPTIMIZATION_PLAN 明确对应关系 |

**核心建议**：优先完成 P0 三项（README、主清单标注、主策略标注），再按 P1 做合并与收敛，最后再做目录重组与长期维护机制。这样在不影响现有协作的前提下，逐步把“分析/”变成可检索、可执行、可维护的单一知识入口。
